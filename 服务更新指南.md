# 服务更新指南

代码修改后，根据不同环境选择对应的更新方式。

---

## 本地开发环境

### 方式 A：dotnet run（单个服务）

`dotnet run` 支持热重载，修改代码后自动生效，无需手动重启。

```powershell
dotnet run --project src/Services/AuthService
```

### 方式 B：脚本启动（所有服务）

需要先停止再重新启动：

```powershell
.\stop-all.ps1
.\start-all.ps1
```

### 方式 C：Docker 全量部署

重新构建并启动：

```powershell
cd docker
docker compose up --build -d
```

如果只改了某个服务，可以单独重建：

```powershell
docker compose up --build -d auth-service
```

---

## 生产服务器

### 方式 A：使用部署脚本（推荐）

```bash
cd /opt/MyYiPix
git pull
cd docker
./deploy.sh
```

部署脚本会自动分批重建和重启所有服务，适合全量更新。

### 方式 B：直接 docker compose

```bash
cd /opt/MyYiPix
git pull
cd docker
docker compose -f docker-compose.prod.yml up --build -d
```

### 方式 C：只更新某个服务（最快）

如果只修改了某个服务的代码（如 AuthService），无需重建全部：

```bash
cd /opt/MyYiPix
git pull
cd docker
docker compose -f docker-compose.prod.yml up --build -d auth-service
```

> 统一 Dockerfile 的 build stage 缓存会被复用，只重新编译改动的项目，速度很快。

## 基础设施（一般不需要更新）

PostgreSQL、Redis、RabbitMQ 是第三方镜像，与业务代码无关。除非需要升级版本，否则不需要重建：

```bash
# 仅在需要升级基础设施镜像版本时执行
docker compose -f docker-compose.prod.yml pull postgres redis rabbitmq
docker compose -f docker-compose.prod.yml up -d postgres redis rabbitmq
```

> ⚠️ 升级 PostgreSQL 大版本（如 16→17）需要先备份数据库，升级后可能需要数据迁移。

## 验证更新

```bash
# 查看容器状态，确认服务已重启
docker compose -f docker-compose.prod.yml ps

# 查看某个服务的日志
docker compose -f docker-compose.prod.yml logs -f auth-service

# 测试接口
curl https://yipixapi.lilikk.com/health
```
